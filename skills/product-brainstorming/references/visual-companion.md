# Visual Companion

Use the browser companion only when seeing the idea will help the user make a decision faster than reading about it.

## Use It For

- Wireframes, layout options, and navigation structure
- Screen flow diagrams or state diagrams
- Side-by-side redesign comparisons
- Visual hierarchy, polish, and interaction concepts

## Do Not Use It For

- Requirement gathering
- Scope decisions
- Product strategy tradeoffs that are easier to discuss in prose
- Technical architecture or API decisions that do not benefit from visuals

## Usage Rules

- Offer it once, then proceed in text if the user declines.
- Make the offer its own message with no additional content.
- Decide per question, not per session.
- Keep each visual focused on one decision.
- Prefer 2-4 options, not giant galleries.
- If the conversation moves back to text-only questions, stop using the browser until a new visual question appears.

## Offer Text

> Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. This feature is still new and can be token-intensive. Want to try it? (Requires opening a local URL)

## Practical Guidance

- Start with low-fidelity visuals unless the decision depends on polish.
- Use real content when placeholder content would hide the actual product question.
- Pair the visual with a short terminal summary of what the user is choosing.
- Treat the terminal response as the primary source of truth, even if the browser records selections.
