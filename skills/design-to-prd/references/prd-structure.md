# PRD Structure

Use the PRD to capture product intent in a form that planning and implementation can trust.

## Section Intent

- `Summary`: the shortest useful description of the feature or change
- `Goal`: what outcome the feature is meant to create
- `Users / Audience`: who the feature is for when that can be inferred safely
- `Source Inputs`: which designs, notes, or specs informed the document
- `Core User Flows`: the main journeys and transitions between screens or states
- `Functional Requirements`: what the system must do
- `Product and Behavioral Rules`: constraints, logic, and interaction rules
- `States, Edge Cases, and Failure Handling`: what happens when inputs are invalid, data is missing, or the user is between normal states
- `Dependencies and Integration Notes`: backend, content, auth, analytics, or platform assumptions that affect delivery
- `Out of Scope`: explicit boundaries that reduce future drift
- `Open Questions / Assumptions`: what still needs confirmation
- `Validation Expectations`: what should be true when the feature is implemented correctly

Keep the document decision-oriented rather than screen-by-screen descriptive when possible.
