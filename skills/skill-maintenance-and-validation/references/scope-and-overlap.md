# Scope And Overlap

Use this reference when deciding whether to update a skill, split a new one out, or keep work in one owner.

## Prefer Updating An Existing Skill When

- the new behavior is already part of the skill's natural workflow
- the examples and rules strengthen the same core job
- creating another skill would mostly duplicate triggering language

## Prefer Creating A New Skill When

- the work repeatedly appears as a distinct second phase
- the completion criteria are materially different
- the new workflow would otherwise bloat the original skill
- routing will become clearer, not muddier

## Prefer Explicit Handoffs When

- one skill owns broad implementation
- another owns a narrower polish, verification, or maintenance phase
- the boundary is easy to describe in one sentence

## Smell Checks

Be careful if:

- two skills trigger on nearly identical language
- one skill is becoming a catch-all for unrelated jobs
- references are growing because the core boundary is unclear
- the skill name and description are broader than the actual workflow
